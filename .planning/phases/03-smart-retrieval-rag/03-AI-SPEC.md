## 1. System Classification

**System Type:** RAG

**Description:**
A specialized Legal RAG system for Vietnamese law. It classifies user queries into legal domains, performs hybrid search (Vector + FTS5) with domain-specific boosting, and generates answers using Gemini. The system must ensure high precision by grounding answers in retrieved document chunks and providing strict, formal citations (IRAC framework).

**Critical Failure Modes:**
1. **Citation Hallucination:** Citing a non-existent Law, Article, or Clause, or misattributing a provision.
2. **Domain Misclassification:** Improperly filtering out relevant laws due to incorrect query categorization.
3. **Context Mixing:** Providing an answer that combines contradictory provisions from different types of legal documents (e.g., an expired Decree vs. a new Law).
4. **Formatting Non-compliance:** Failing to follow the strict "Theo Khoản X, Điều Y..." citation format required for legal authority.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Legal (Vietnamese Jurisprudence)

**User Population:** General public and legal practitioners seeking quick references to Vietnamese law.

**Stakes Level:** High (Legal advice/information requires absolute accuracy to avoid regulatory or personal risk).

**Output Consequence:** Users may make decisions based on the provided information (e.g., filing a complaint, signing a contract). Accuracy is paramount.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|-------------------|--------|--------|
| **Citation Accuracy** | Citations match the exact Law/Decree/Circular number and Article/Clause. | Incorrect document numbers or Article/Clause references. | Critical | Law Practice Standard |
| **Terminology** | Uses correct legal terms (e.g., "nguyên đơn", "bị đơn", "văn bản quy phạm pháp luật"). | Uses colloquial or imprecise language for legal concepts. | Medium | Legal Lexicon |
| **Hierarchy of Law** | Prioritizes Laws over Decrees, and Decrees over Circulars when conflicts occur. | Treats all documents with equal weight regardless of legal hierarchy. | High | Constitutional Law |

### Known Failure Modes in This Domain
- **Temporal Confusion:** Retrieving expired (hết hiệu lực) documents because they contain keyword matches.
- **Granularity Loss:** Providing a general Law citation when a specific Decree/Circular Article is the actual governing rule.

### Regulatory / Compliance Context
- **Disclaimer Requirement:** Must state that the AI is not a substitute for professional legal counsel.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Legal Analyst / Lawyer | Dataset labeling, rubric calibration for "Citation Accuracy" and "Hierarchy of Law". |

## 2. Framework Decision

**Selected Framework:** LlamaIndex

**Version:** `v0.10.x`

**Rationale:**
LlamaIndex is the industry leader for production-grade RAG systems where retrieval quality is the primary goal. Its "batteries-included" approach to complex retrieval strategies (Recursive Retrieval, Hybrid Search, and Reranking) fits the high-precision requirements of legal research better than general-purpose frameworks like LangChain.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain | Higher abstraction overhead; while flexible, it requires more manual boilerplate to achieve the same RAG precision levels as LlamaIndex. |
| Google ADK | Primarily optimized for Gemini-only workflows; LlamaIndex allows better integration with our existing local `vietnamese-sbert` embedding pipeline. |

**Vendor Lock-In Accepted:** Partial. LlamaIndex is model-agnostic, but we are committing to its internal abstractions for indexing and querying.

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation
```bash
pip install llama-index llama-index-llms-gemini llama-index-embeddings-huggingface llama-index-vector-stores-qdrant
```

### Core Imports
```python
from llama_index.core import VectorStoreIndex, StorageContext, Settings
from llama_index.llms.gemini import Gemini
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.vector_stores.qdrant import QdrantVectorStore
```

### Entry Point Pattern
```python
# Configure Settings
Settings.llm = Gemini(model="models/gemini-2.0-flash", api_key=os.environ["GEMINI_API_KEY"])
Settings.embed_model = HuggingFaceEmbedding(model_name="keepitreal/vietnamese-sbert")

# Connect to existing Qdrant
vector_store = QdrantVectorStore(client=qdrant_client, collection_name="legal_vectors")
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# Load index
index = VectorStoreIndex.from_vector_store(vector_store, storage_context=storage_context)
query_engine = index.as_query_engine(similarity_top_k=5)
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **QueryEngine** | End-to-end interface for querying an index. | For standard retrieval + generation. |
| **NodePostprocessor** | A step that modifies/filters nodes after retrieval but before generation. | For domain-based filtering or reranking. |
| **ResponseSynthesizer** | Component that generates the final answer from retrieved nodes. | For implementing IRAC-style custom prompts. |

### Common Pitfalls
1. **Metadata Filter Syntax:** Qdrant filters in LlamaIndex must strictly match the `MetadataFilters` schema; using raw Qdrant filters can bypass framework optimizations.
2. **Gemini Rate Limits:** High-frequency classification + RAG can trigger Gemini rate limits; implement exponential backoff.

## 4. Implementation Guidance

**Model Configuration:**
- Classifier: `gemini-2.0-flash` (low temperature 0.1 for deterministic labels).
- Generator: `gemini-2.0-flash` (temperature 0.3 for slightly more natural explanation while keeping citations strict).

**Core Pattern:**
- **Hybrid Search + RRF:** We will leverage the existing FTS5 + Qdrant logic, using LlamaIndex's `CustomRetriever` to orchestrate the hybrid flow and RRF fusion.
- **Multi-Label Classifier:** A pre-query step that outputs a list of domain tags. These tags are passed to the `MetadataFilters` of the retriever.

**Tool Use:**
- **Qdrant:** Persistent vector storage.
- **SQLite:** Metadata and FTS5 storage.

**State Management:**
- Stateless per-query, but uses the persistent `indexing_status` and metadata databases established in Phases 1 & 2.

**Context Window Strategy:**
- Limit to Top 5-8 chunks (300-500 tokens each) to stay well within Gemini's context window while avoiding the "lost in the middle" effect.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs.

### Structured Outputs with Pydantic
The Classifier must return a structured JSON object to ensure the backend can reliably apply filters.

```python
from pydantic import BaseModel, Field
from typing import List

class QueryClassification(BaseModel):
    domains: List[str] = Field(description="List of legal domains relevant to the query")
    confidence: float = Field(description="Confidence score for the classification")
    is_explicit_filter: bool = Field(description="True if user explicitly requested a specific law")
```

### Async-First Design
Use `aquery` for non-blocking queries to the LLM, ensuring the FastAPI backend remains responsive.

### Prompt Engineering Discipline
- **System Prompt:** Define the agent as a "Senior Vietnamese Legal Expert".
- **Few-shot:** Provide 3-5 examples of query-to-domain mapping in the classifier prompt.
- **Citations:** Use "In-context Learning" by providing clear examples of how to cite the retrieved chunks.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| **Faithfulness** | 1-5 (No hallucination) | LLM Judge (RAGAS) | Critical |
| **Answer Relevance** | 1-5 (Directly answers query) | LLM Judge (RAGAS) | High |
| **Citation Format** | Pass/Fail (Matches regex) | Code-based (Regex) | Critical |
| **Domain Precision** | Pass/Fail (Correct tags) | LLM Judge vs Ground Truth | Medium |

### Eval Tooling

**Primary Tool:** RAGAS + Arize Phoenix

**Setup:**
```bash
pip install ragas arize-phoenix
```

**CI/CD Integration:**
Run a script `scripts/eval_pipeline.py` that calculates RAGAS scores against the reference dataset and fails the build if `faithfulness < 0.85`.

### Reference Dataset

**Size:** 20 examples to start.

**Composition:**
- 10 "Happy Path" legal questions with clear domain labels.
- 5 "Cross-domain" questions (e.g., Tax + Business).
- 5 "Edge cases" (queries about laws not in the dataset or ambiguous wording).

**Labeling:**
Labeled by a human (Developer/Legal POC) with ground truth citations and expected domain tags.

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix

**Key Metrics to Track:**
1. **Average Latency:** Goal < 5s per RAG call.
2. **Classification Confidence:** Monitoring for "General" label frequency.
3. **Token Usage:** Tracking costs per session.

**Alert Thresholds:**
- Latency > 10s for 5 consecutive calls.
- Gemini API error rate > 5%.

**Smart Sampling Strategy:**
Review any trace where "Classification Confidence" < 0.4 or where RAGAS Faithfulness score in periodic sampling falls below 0.7.

# Phase 6.2: DeepSeek API Classifier Integration - Context

**Gathered:** 2026-04-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the Qwen-based classifier with a DeepSeek-based implementation to resolve token errors and improve classification precision. This involves creating a dedicated DeepSeek client, optimizing the Vietnamese legal prompt, and configuring a robust failover to Gemini.

</domain>

<decisions>
## Implementation Decisions

### API & Model Selection
- **D-01: Official Provider:** Use the official DeepSeek API endpoint (`https://api.deepseek.com/v1`).
- **D-02: Model Variant:** Use `deepseek-chat` (DeepSeek-V3) as it offers the best balance of speed, cost, and classification performance for Vietnamese text.
- **D-03: Primary Role:** DeepSeek will become the primary provider for the `LegalQueryClassifier`.

### Failover & Resilience
- **D-04: Gemini Fallback:** If DeepSeek returns an error, exceeds rate limits, or times out, the system must fallback to **Gemini-2.0-Flash** (the established stable provider).
- **D-05: Qwen De-prioritization:** Qwen (DashScope/Ollama) will be removed or moved to the end of the failover chain due to current reliability issues.

### Optimization & Integration
- **D-06: Vietnamese Prompt Tuning:** The system prompt in `core/classifier.py` will be refined to leverage DeepSeek's strong performance in Vietnamese legal terminology.
- **D-07: Client Pattern:** Implement `DeepSeekClient` in `tools/deepseek_client.py` following the `BaseLLMClient` contract (async `generate_content_async`).

### the agent's Discretion
- Exact timeout settings for DeepSeek API calls.
- Specific refinements to the Vietnamese system prompt strings.
- Whether to keep the Qwen client code in `tools/` for future use or remove it entirely (recommended: keep code but remove from active classifier chain).

</decisions>
